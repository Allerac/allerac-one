# Native iOS Client

## Status

Direction document. Implementation has not started.

The production Control API, scoped device keys, and Cloudflare path policy already
provide the server boundary required by a native iOS client. The remaining work is
an Apple-platform client and its build, signing, and installation workflow.

## Product Role

The iOS app should be a first-class Allerac client, not a WebView wrapper. Its first
release should provide a reliable personal interface to an existing Allerac
deployment while keeping account administration and API-key creation in the web UI.

The initial target device is an iPhone 13 Pro. The architecture must remain usable
on other supported iPhones without device-specific assumptions.

## Proposed Stack

- **SwiftUI** for application structure and interface.
- **URLSession** with async/await for Control API requests.
- **Keychain** for the deployment URL's API key.
- **Speech** for user-approved speech recognition.
- **AVFoundation** for audio playback and session management.
- **App Intents** only after the core client is stable.

No provider credential belongs in the app. OpenAI, Anthropic, GitHub, and other
service credentials remain on the Allerac deployment; the client receives only a
scoped Allerac API key.

## Production Connection

The production client uses:

```text
Base URL: https://app.allerac.ai
API root: /api/v1
Authorization: Bearer alr_live_...
```

Cloudflare must continue to pass `/api/v1/*` without an interactive Access login.
Allerac remains responsible for client authentication and authorization. Each iOS
installation receives its own revocable key.

Recommended MVP scopes:

```text
profile:read
capabilities:read
chat:read
chat:write
```

Additional domain scopes are added only when the corresponding screen is delivered.

## MVP User Flow

1. Enter or scan a deployment base URL and device API key.
2. Validate the deployment with `/api/v1/version`, `/api/v1/me`, and
   `/api/v1/capabilities`.
3. Create a conversation or select an existing one.
4. Send text through `/api/v1/conversations/:id/messages`.
5. Display the assistant response and conversation history.
6. Optionally dictate input and play speech generated by `/api/v1/speech`.
7. Revoke the device key from the Allerac web UI if the phone is lost or replaced.

## Client Boundaries

The app may persist:

- the deployment URL;
- a Keychain reference to the API key;
- the current conversation identifier;
- non-sensitive interface preferences;
- a bounded local cache of responses when offline support is introduced.

The app must not persist the API key in source code, `UserDefaults`, logs, crash
reports, screenshots, or repository configuration. HTTP logs must redact
`Authorization` headers.

## Voice

Voice is an incremental capability, not a prerequisite for text chat. The app must
request microphone and speech-recognition permission only when the user enables or
invokes voice input. Denial must leave text chat fully functional.

The first voice flow can mirror the Android robot:

1. capture or transcribe user speech on the device;
2. send the resulting text to the Control API;
3. request server-side speech synthesis when available;
4. play the result through an `AVAudioSession` configured for the current context.

Always-on listening, background capture, and wake words are explicitly outside the
MVP because they introduce significant privacy, battery, and platform constraints.

## Build and Distribution Boundary

Repository work can be designed and reviewed from any development machine, but a
native iOS build must be compiled and signed with Xcode on macOS. Development does
not require publication in the App Store:

- a developer can install a signed build directly from Xcode on a connected iPhone;
- TestFlight can be introduced when repeatable external beta distribution is needed;
- App Store review is a later product-distribution decision, not an MVP dependency.

The project must never commit signing certificates, private keys, provisioning
profiles, Apple credentials, or exported Keychain material.

## Relationship to Existing Clients

The iOS app reuses the same Control API and per-device authorization model as the
Android robot. Shared API contracts and fixtures may be reused, but platform UI,
secure storage, audio, permissions, and lifecycle code should remain native.

See [Mobile App and Home Robot Clients](mobile-and-robot-clients.md) and
[ADR 0003](decisions/0003-expose-control-api-through-cloudflare-path-policy.md).

## Architecture Exit Criteria

The architecture is validated when a signed development build on a physical iPhone:

- reaches the production API through Cloudflare;
- authenticates with an iOS-specific scoped key;
- reports the deployed release and capabilities;
- creates and resumes a conversation;
- sends a message and renders the response;
- keeps the key out of logs and ordinary application storage;
- continues to support text chat when voice permissions are denied.
